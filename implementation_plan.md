# Implementation Plan - College Society Web Application

This document outlines the architecture, database schema, system flows, and file organization for the College Society web application.

## Technical Architecture & Database Design

### 1. Relational Database Schema (PostgreSQL)

We will define 8 primary models inside `models/`:

#### A. `roles` & `users` (Role-Based Access Control)
- **`Role`**:
  - `id`: Integer (PK)
  - `name`: String (Unique) — `Super Admin`, `Content Admin`, `Gallery Admin`, `Moderator`, `User`
  - `permissions`: JSON (E.g. `{"manage_journals": true, "manage_events": true, "manage_gallery": true, "manage_users": true, "view_logs": true, "delete_logs": true}`)
- **`User`**:
  - `id`: Integer (PK)
  - `email`: String (Unique, Indexed)
  - `hashed_password`: String
  - `full_name`: String
  - `role_id`: Integer (FK -> `roles.id`)
  - `is_active`: Boolean (Default: True)
  - `created_at`: DateTime
  - `updated_at`: DateTime

#### B. `journals` (Moderation & Workflow)
- **`Journal`**:
  - `id`: Integer (PK)
  - `title`: String
  - `abstract`: Text
  - `content_url`: String (Cloudinary URL for PDF or rich document)
  - `author_id`: Integer (FK -> `users.id`)
  - `tags`: String (Comma separated)
  - `category`: String
  - `status`: String (Default: `Pending`, Options: `Pending`, `Approved`, `Rejected`)
  - `moderator_comment`: Text (Admin rejection comments)
  - `is_archived`: Boolean (Default: False)
  - `is_deleted`: Boolean (Default: False)
  - `deleted_at`: DateTime (Nullable)
  - `created_at`: DateTime
  - `updated_at`: DateTime

#### C. `events`
- **`Event`**:
  - `id`: Integer (PK)
  - `title`: String
  - `description`: Text
  - `date`: DateTime
  - `location`: String
  - `banner_url`: String (Cloudinary URL)
  - `registration_url`: String (Nullable)
  - `is_archived`: Boolean (Default: False)
  - `is_deleted`: Boolean (Default: False)
  - `deleted_at`: DateTime (Nullable)
  - `created_at`: DateTime

#### D. `gallery_items`
- **`GalleryItem`**:
  - `id`: Integer (PK)
  - `title`: String
  - `image_url`: String (Cloudinary URL)
  - `category`: String
  - `is_archived`: Boolean (Default: False)
  - `is_deleted`: Boolean (Default: False)
  - `deleted_at`: DateTime (Nullable)
  - `created_at`: DateTime

#### E. `news_resources`
- **`NewsResource`**:
  - `id`: Integer (PK)
  - `title`: String
  - `type`: String (`News` or `Resource`)
  - `content`: Text (Optional)
  - `file_url`: String (Nullable, Cloudinary URL for PDFs)
  - `is_archived`: Boolean (Default: False)
  - `is_deleted`: Boolean (Default: False)
  - `deleted_at`: DateTime (Nullable)
  - `created_at`: DateTime

#### F. `audit_logs` (User and Admin Tracking)
- **`AuditLog`**:
  - `id`: Integer (PK)
  - `user_id`: Integer (FK -> `users.id`, Nullable for public/guest actions)
  - `user_email`: String (For quick viewing if user deleted)
  - `action`: String (E.g. `LOGIN`, `CREATE_JOURNAL`, `APPROVE_JOURNAL`, `DELETE_EVENT`, `RESTORE_EVENT`, `DELETE_LOG`)
  - `target_type`: String (E.g. `journal`, `event`, `gallery`, `user`, `log`)
  - `target_id`: Integer (Nullable)
  - `details`: String (Detailed description / JSON parameters)
  - `created_at`: DateTime

#### G. `archive_settings` (Dynamic Configuration)
- **`ArchiveSetting`**:
  - `id`: Integer (PK)
  - `category`: String (Unique - e.g., `journals`, `events`, `gallery`, `news`)
  - `duration_days`: Integer (Number of days after which content is auto-archived. If <=0, auto-archive is disabled for this category)

---

### 2. Auto Archive System
We will integrate a lightweight background scheduler (using **APScheduler** in Python or a custom async task scheduler embedded in FastAPI's startup event loop).
- Every 1 hour, the job checks `archive_settings` rules.
- If an item (Journal, Completed Event, Gallery, News) exceeds its configured age (e.g. `created_at < current_time - archive_days`), `is_archived` is automatically set to `True`.
- Archived items are filtered out from public read endpoints but remain visible in the Admin Portal under the "Archived" view, with an option to manually restore them to the public page.

---

### 3. Soft Delete Recycle Bin System
- **Soft Deletion**: Any delete operation on Journals, Events, Gallery, or News sets `is_deleted = True` and `deleted_at = datetime.utcnow()`. It will immediately disappear from the public website and the main admin management panels.
- **Recycle Bin View**: Admins can see these items under a dedicated **Recycle Bin** sub-panel.
- **Restore / Force Delete**:
  - Clicking **Restore** sets `is_deleted = False` and `deleted_at = None`.
  - Clicking **Delete Permanently** triggers a hard database delete and (optionally) deletes the file from Cloudinary using its public ID.
- **Automatic Hard Delete Cleanup**:
  - A background task runs daily. It query all records across the four content models where `is_deleted = True` and `deleted_at < current_time - 15 days`.
  - It permanently deletes these stale records from the database and purges their assets from Cloudinary.

---

### 4. Admin Audit Logs Portal
- Every key modification (CRUD, approval/rejections, logins, and log purges) creates an entry in `audit_logs` table.
- We will build an **Admin Logs Dashboard** for Super Admins.
- **Features**:
  - **Search & Filter**: Search logs by operator email, target category, action name, or date range.
  - **Pagination**: Handle large volumes of log files gracefully.
  - **Delete Logs**: Under Super Admin permission, allows deleting specific log items or purging log history (e.g. older than 30 days) to prevent log tables from bloating. All log deletion actions themselves will write a permanent log entry that cannot be deleted to ensure an audit trail.

## Verification Plan

### Automated Tests
1. **FastAPI Unit Tests**: We will write backend tests using `pytest` and `httpx.AsyncClient` focusing on:
   - Registration, login, and JWT token rotation.
   - RBAC rules validation (e.g. User cannot approve a journal, Gallery Admin cannot delete an event).
   - Soft-delete operations and recycle bin restoration.
   - Auto-archive logic triggered via service functions.
   - Audit logs capture on database CRUD events.
2. **Execution Commands**:
   ```bash
   cd backend
   pytest
   ```

### Manual Verification
1. **Interactive Swagger Docs**: We can verify all endpoints through FastAPI's `/docs` page.
2. **Frontend UI Walkthrough**:
   - Check landing page animations, grids, and responsiveness.
   - Register a new user account, log in, submit a test journal.
   - Log in as the Super Admin, approve the journal, verify it appears publicly.
   - Trigger a soft delete on a gallery item, verify it disappears from the gallery, find it in the Recycle Bin, restore it, and confirm it reappears.
   - View actions inside the Action Logs dashboard, search by keyword, and delete logs.
